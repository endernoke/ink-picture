# Development

This document covers the internal architecture and implementation details of ink-picture.

## Architecture overview

ink-picture renders images using two strategies: **text-based protocols** that produce Ink-compatible output, and **graphical protocols** that write escape sequences directly to `stdout`.

### Text-based protocols

Text-based protocols (`ascii`, `braille`, `halfBlock`) generate plain strings of visible characters and ANSI color codes. These are rendered as Ink `<Text>` children within the component tree, so they participate in Ink's normal layout and rendering cycle. This means:

- They are automatically clipped by Ink's layout engine if they overflow their container.
- They are cleared and re-rendered each frame alongside the rest of the Ink output.
- They don't require any coordination with the terminal cursor position.

### Graphical protocols

Graphical protocols (`sixel`, `kitty`, `iterm2`) write escape sequences directly to `stdout` outside of Ink's normal rendering. Because Ink doesn't know about these writes, several challenges arise:

1. The terminal cursor must be moved to the target position before writing the image data.
   `useDirectRenderer` (used by Sixel and iTerm2) and the `KittyImage` component use ANSI cursor up sequence to move the cursor to the target position, and save/restore sequences to return the cursor to its original position after painting.

since the terminal cursor cannot be positioned outside the viewport, graphical protocol output is skipped when the image is not fully visible, otherwise it would cause image dislocation.

2. images must be repainted at intervals.
   By default, Ink clears the terminal buffer before rendering each frame, which removes any sixel and iterm2 images even if their positions are unchanged. Images will be cleared by Ink on **every re-render in the component tree**, and clearing happens without the image component knowing or re-rendering.

As a result, sixel and iterm2 images have to be redrawn in short intervals in order to not get wiped by every app update. The `paintIntervalMs` config controls this rate. Each tick, the component checks the element's current position and re-writes the image data to `stdout` at the correct location.

Note that this does not apply to the kitty renderer because kitty graphics do not share the same buffer as text by design. Writing spaces over kitty images does not clear them.

3. Active polling is required to respond to layout changes.
   Element positions can be computed after each React render of a component, but layout changes can occur without triggering a re-render (e.g. parent dimensions change). Images may therefore be dislocated immediately after a layout change. To solve this, a polling loop (at `pollIntervalMs`) traverses the Yoga layout tree to detect and propagate position changes.

### `useDirectRenderer` (for Sixel and iTerm2)

Sixel and iTerm2 images share the same rendering mechanism via `useDirectRenderer`:

- On each tick, it writes the image output to `stdout` at the element's position using ANSI cursor movement sequences.
- On unmount, it fills the image's bounding box with background-colored spaces (or plain spaces) to clean up the area.
- `useBackgroundColor` walks up the React element tree to find the nearest ancestor with a `backgroundColor` style set, so clean-up spaces match the surrounding background.

### Kitty protocol

Kitty uses a different approach:

- Images are first _uploaded_ to the terminal's image cache via transmit escape sequences, so the image data is only transmitted once per component lifecycle.
- Images are _placed_ at cursor positions on each re-render with the same placement ID, which would also automatically delete the old placement, achieving flicker-free movement.
- On unmount, a deletion command removes the image from the terminal's cache.

Kitty image IDs are generated by a singleton `KittyIdGenerator` that reuses freed IDs.

## Terminal capability detection

`queryTerminal` (in `src/utils/queryTerminal.ts`) sends a batch of ANSI escape sequence queries at once to the terminal when the `InkPictureProvider` mounts (usually on startup). The device attributes response sent at the end serves as a sentinel to signal response completion. A 1000ms timeout prevents indefinite waiting.

During detection, `stdin.push` is temporarily overridden to intercept the escape sequence responses, preventing them from being picked up by Ink's `useStdin`, `useInput`, or `usePaste` hooks. Any user input received during this window is pushed back into the `stdin` stream after detection completes.

## Image caching

In order to reduce flickering when an image switches protocol, an LRU cache `ImageCache` for `Jimp` image objects is used. Cached images are cloned on retrieval to avoid mutation. Caching is disabled when `cacheSize` is set to `0`.

## Testing

````bash
```bash
pnpm format
pnpm lint [--fix]

pnpm test              # Full typecheck + lint + unit + browser tests
pnpm test:unit         # vitest tests
pnpm test:browser      # Browser tests using playwright with xterm.js
````

- Unit tests (`tests/vitest/`) cover renderers, utilities, and hooks that don't require terminal escape sequence interpretation.
- Browser tests (`tests/playwright/`) spin up a real xterm.js terminal in a headless browser to test graphical protocol rendering and terminal interaction end-to-end.
